# Ezer V3.1 — Technical Architecture & Schema Guide
*For Developers*

---

## System Overview

Ezer is a deterministic, zero-LLM insight engine. At runtime it loads two JSON files per user, joins them in memory, and produces structured insight objects. No database. No inference. No external calls.

```
Tier 1 (Product Library JSON)
         +
Tier 2 (User Schedule JSON)
         ↓
    Insight Engine
         ↓
  Insight Objects + Summary + Timeline + Renewal Guidance
```

---

## Data Contracts

### Tier 1 — Product Library

One file per insurance product. Shared across all users of that product.

**Key fields the engine reads:**

| Field | Path | Used For |
|-------|------|----------|
| Product ID | `_schema_meta.product_id` | Routing product-specific logic |
| Moratorium months | `moratorium_period.months` | Legal protection check |
| Specific disease list | `waiting_periods.specific_disease_waiting.illnesses` | Plain-English timeline labels |
| Optional addons | `optional_addons_available` | Renewal guidance gap detection |
| Consumables note | `consumables_coverage.annexure_i_excluded` | Protector Rider gap check |

**File naming:** `product_library_{insurer}_{product_name}.json`

---

### Tier 2 — User Schedule

One file per policy number. Never shared. Contains only user-specific data.

**Key fields the engine reads:**

| Field | Path | Used For |
|-------|------|----------|
| Product ID | `_schema_meta.product_id` | Loads matching Tier 1 file |
| SI blocks | `waiting_periods.si_blocks[]` | Waiting period checks, maturity, clean SI |
| SI enhanced flag | `si_blocks[].si_enhanced` | Fortress vs enhancement detection |
| Approximate expiry | `si_blocks[].*.approximate_expiry` | Timeline milestone generation |
| Declared PEDs | `declared_peds[]` | PED waiting, quality flags |
| ezer_flags | `declared_peds[].ezer_flags[]` | Action outputs, quality flags |
| Flag status | `ezer_flags[].status` | RESOLVED flags are skipped |
| Schedule integrity | `waiting_periods.schedule_integrity_flags[]` | Error detection, action letters |
| Addons | `addons` or `addons_active` | Feature checks, renewal guidance |
| Rider ineligibility | `addons.rider_ineligible_due_to_age` | Senior exception for MATURE check |
| Months coverage | `policy_meta.months_continuous_coverage` | Moratorium check |
| Plus/Multiplier status | `sum_insured.plus_benefit_status` | Clean SI accrual gate |

**File naming:** `user_schedule_{policy_number}_{insured_name}.json`

## The Evaluative Matrix — compute_summary() Logic

```python
# Step 1: Segment insights by type
actions   = [i for i in insights if i["type"] == "ACTION"]
warnings  = [i for i in insights if i["type"] == "WARNING"]
high_warn = [i for i in warnings  if i["priority"] == "HIGH"]

# Step 2: Compute Clean SI ratio (0.0 to 1.0)
clean_pct = compute_clean_si_ratio(lib, sch)

# Step 3: Derive Strength — Actions take absolute priority
if actions >= 1:
    strength = "WEAK"                                          # Action override — always
elif clean_pct >= 0.75 and high_warn == 0:
    strength = "STRONG"                                        # Fortress clean, no HIGH risks
elif clean_pct >= 0.75 and high_warn > 0:
    strength = "MODERATE"                                      # Fortress clean, clocks cooking
elif high_warn >= 1:
    strength = "MODERATE"                                      # Significant risks present
else:
    strength = "STRONG"                                        # No risks, no actions

# Step 4: Derive Maturity — purely structural
maturity = derive_maturity_status(lib, sch)
# DEVELOPING → any base/fortress block has running waiting period
# MATURING   → base clean, enhanced blocks still cooking
# MATURE     → all clocks served + critical riders active (or ineligible_due_to_age)

# Step 5: Derive Next Best Action
if actions:
    NBA = first HIGH ACTION → strip "Action Required — " → append "today"
elif HIGH warnings:
    NBA = first HIGH WARNING → use suggested_action field if present
```

## Fortress Detection Logic

The engine identifies fortress (base) vs enhancement blocks at runtime. It does not rely on hardcoded SI amounts.

**Detection hierarchy (in order):**
1. `si_enhanced == True` in block JSON → enhancement
2. `"enhanced"` appears in `block_id` string → enhancement
3. Block SI is smaller than the largest block SI in the schedule → enhancement
4. Everything else → fortress

**Implication:** The largest SI block is always treated as the fortress. This is correct for all current supported products. When adding new products with different SI structures, verify the detection logic holds.

## Clean SI Ratio — compute_clean_si_ratio()

```python
# For each SI block:
if spec_waiting != "running" AND ped_waiting != "running":
    clean_si += block_si
total_si += block_si

# For Optima Secure bonus:
if plus_benefit_status == "fully_accrued":
    clean_si += plus_si        # Only if locked in
total_si += plus_si
clean_si += secure_benefit_si  # Always clean
total_si += secure_benefit_si

# For Optima Restore bonus:
if multiplier_benefit_status == "fully_accrued":
    clean_si += multiplier_si  # Only if locked in
total_si += multiplier_si

return clean_si / total_si     # Float 0.0 to 1.0
```

**The accrual gate is mandatory.** A partially accrued bonus is potential SI, not clean SI. Including it prematurely would overstate coverage strength.

## Timeline Engine — generate_evolution_timeline()

**Input:** All `si_blocks` in Tier 2 with `approximate_expiry` fields.

**Grouping logic:**
```python
milestones = defaultdict(list)   # key = datetime object

for each block:
    if spec_waiting == "running" and expiry exists and expiry > today:
        milestones[datetime(expiry)].append(event_string)
    if ped_waiting == "running" and expiry exists and expiry > today:
        milestones[datetime(expiry)].append(event_string_with_ped_names)

# Sort by datetime object — not string sort
sorted_milestones = sorted(milestones.keys())
```

**Why datetime sort matters:**
String sort would order "February 2028" before "January 2029" incorrectly if months alphabetize oddly. Datetime objects sort correctly across year boundaries.

**Empty state:** If no future milestones exist (all clocks served), the section is omitted entirely. The user does not see an empty block.

**PED name injection:** The engine cross-references `declared_peds[]` to replace generic "Declared PED(s)" with the actual condition name where available.
