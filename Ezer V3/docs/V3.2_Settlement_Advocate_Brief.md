## What Was Built

Three new components added to the Ezer engine:

**`data/settlements/settlement_GPS_main.json`**
Ground truth file for Gupta Prasad Satapathy's Urolift claim (CCN RR-HS25-15050055). Human-curated from HDFC ERGO settlement letter. 23 line items + 6 hospital flags. Powers the audit engine deterministically.

**`engine/audit_engine.py`**
Deterministic adjudication engine. Reads settlement JSON, validates schema via Pydantic, computes Total Asymmetry Payload, runs rider simulation, prepares talking memo data.

**`engine/symmetry_report.py`**
Report generator. Ingests audit output and user schedule. Determines route (A/B/C) from Tier 2 data. Generates three output documents. No user input required.

**`tests/test_routing.py`**
Unit test for the Decision Gate. Three scenarios, 3/3 pass. Route A/B/C all deterministic.

## The Asymmetry Payload — What It Means

```
Layer 1 — Invisible Loss (Hospital Flags):   ₹95,561.81
Layer 2 — Visible Loss (Insurer Deductions): ₹ 9,347.00
Total Asymmetry Payload:                     ₹1,04,908.81
```

**Invisible Loss** — charges the hospital billed and the insurer paid without auditing. The patient never sees these flagged. Ezer surfaces them.

**Visible Loss** — charges the insurer deducted. The patient sees the deduction but not the reason. Ezer classifies each deduction and identifies which are recoverable.

## The Six Hospital Flag Types

| Flag Type | What It Means |
|-----------|---------------|
| DUPLICATE_CHARGE | Same service billed twice |
| QUANTITY_ANOMALY | Volume inconsistent with procedure (e.g. 21L saline) |
| DEFINITION_REQUIRED | Charge exists but scope undefined (e.g. ₹84k equipment) |
| GHOST_CONSULTATION | Service billed, no clinical basis (dietician on urology) |
| HOSPITAL_FLAG | Generic — unbundling, double-dip, administrative padding |

## The Decision Gate — Three Routes

The `determine_rider_route()` function reads Tier 2 user schedule. No user input. Fully deterministic.

```
Read addons_active.protector_rider
        ↓
    True → Route A (Grievance Mode)
           Rider active. Insurer wrongly deducted.
           Generate formal grievance letter.

    False → Check rider_ineligible_due_to_age.protector_rider
                ↓
            Flag exists → Route B (Gap Awareness Mode)
                          Rider unavailable due to age.
                          Generate honest gap statement.

            No flag → Route C (Renewal Action Mode)
                      Rider available but not purchased.
                      Generate renewal checklist.
```

## Key Architectural Decisions

**Decision 1 — Curated JSON over PDF extraction**
Settlement data is human-curated into JSON — same approach as policy schedules. No LLM extraction. No hallucination risk. Every line item verified against the settlement letter before entry.

**Decision 2 — Route B over Route A for age-ineligible users**
When a rider is informally unavailable due to age, the engine does not suppress the recovery amount — it shows ₹7,017 lost AND explains why recovery at renewal may not be possible. Honest insight over false promise.

**Decision 3 — Hospital flags separate from insurer deductions**
Layer 1 (hospital) and Layer 2 (insurer) are computed and displayed separately. The user understands who caused each loss and who to approach.

**Decision 4 — PII never written to output files**
Patient name and account number are read from JSON for internal processing only. All three output documents contain claim reference and hospital name (institutional identifiers) but no personal identifiers.

**Decision 5 — Pydantic full contract validation**
All enums validated on load. A typo like "HOSPITAL_FLG" raises a ValidationError immediately — the engine refuses to run on invalid data. Silent failures are unacceptable in insurance adjudication.

## What V3.2 Does Not Do (Deliberate Exclusions)

- No Ombudsman filing generator — Route A grievance is Level 1. Ombudsman is Level 3. Premature escalation weakens legitimate cases.
- No PDF extraction — reliability non-negotiable in insurance context.
- No hospital bill pricing database — no standard rates exist in India. Contestable items are identified by existence and appropriateness, not price.
- No automatic claim filing — Ezer informs. User acts.
