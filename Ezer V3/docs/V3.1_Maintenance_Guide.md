## Adding a New Insight

Every insight in Ezer is produced by a `check_` function. Each function takes `lib` (Tier 1) and `sch` (Tier 2) as arguments and returns a list of insight objects.

**Step 1 — Write the check function**

```python
def check_your_new_insight(lib: dict, sch: dict) -> list:
    insights = []

    # Read from Tier 2
    value = sch.get("some_field", {}).get("nested_key", None)
    if not value:
        return insights

    # Optionally cross-reference Tier 1
    product_rule = lib.get("relevant_section", {}).get("rule", "")

    insights.append(insight(
        category="ACTIVE_FEATURE",          # or GAP_FEATURE / WAITING_PERIOD / etc.
        confidence=VERIFIED,                # or EXTRACTED / EXPERIMENTAL
        title="Short label — one line",
        body="Plain-language explanation. What it means. What the user should know.",
        type="INFO",                        # ACTION / WARNING / INFO
        priority="LOW",                     # HIGH / MEDIUM / LOW
        suggested_action="Imperative command for WARNING insights. Empty for INFO."
    ))

    return insights
```

**Step 2 — Register it in generate_insights()**

```python
def generate_insights(schedule: dict) -> list:
    ...
    all_insights += check_your_new_insight(library, schedule)
    ...
```

Add it in the logical order: active features → gaps → waiting periods → PED checks → schedule integrity → action outputs.

**Step 3 — Classify correctly**

| Situation | type | priority |
|-----------|------|----------|
| User must send a letter today | ACTION | HIGH |
| Claim can be denied today | WARNING | HIGH |
| Gap fixable at renewal | WARNING | MEDIUM |
| Awareness, no risk | INFO | LOW |

Never mix problem and solution in one insight. Problem → WARNING. Solution (letter) → ACTION.

**Step 4 — Test against all 4 schedules**

```bash
cd ~/Desktop/Ezer/Ezer\ V3
python3 engine/insight_engine.py
```

Verify the insight appears correctly for the intended user and does not appear for users where it should not trigger.

## Updating the 75% Threshold

The Clean SI threshold is defined in `compute_summary()`:

```python
elif clean_pct >= 0.75 and len(high_warn) == 0:
    strength = "STRONG"
elif clean_pct >= 0.75 and len(high_warn) > 0:
    strength = "MODERATE"
```

To change the threshold (e.g. to 80%):
- Replace `0.75` with `0.80` in both lines
- Run against all schedules and verify the output still reflects product intent
- Document the reason for the change in DECISION.md

**Do not change this without product sign-off.** The 75% threshold was a deliberate product decision. Changing it silently will produce different Strength ratings for existing users without their knowledge.

## Updating Fortress Detection Logic

The engine identifies the base/fortress block vs enhanced blocks in `derive_maturity_status()` and `compute_clean_si_ratio()`:

```python
# Current detection hierarchy
is_enhanced = (
    block.get("si_enhanced", False) or      # Explicit flag in Tier 2
    "enhanced" in block_id.lower() or       # Block ID contains "enhanced"
    block_si < max_si_block                 # Smaller than largest block
)
```

**When to update this:**
- A new product has a structure where the largest block is NOT the fortress
- A new product uses different block naming conventions

**How to update:**
- Add the new detection condition to the same `is_enhanced` expression
- Test against the new product's user schedule before adding to production library
- Do not remove existing conditions — they cover real data already in production

## Adding a Grievance Letter Template

All draft letters are generated by dedicated `draft_*` functions. Each function takes `sch` (Tier 2) and the relevant flag dict as arguments.

**Current templates:**
- `draft_schedule_correction_letter(sch, flag)` — schedule integrity errors
- `draft_ped_clarification_letter(sch, ped)` — PED ICD vagueness

**To add a new letter template:**

```python
def draft_your_new_letter(sch: dict, context: dict) -> str:
    name   = sch.get("policy_meta", {}).get("insured_name", "Insured")
    policy = sch.get("_schema_meta", {}).get("policy_number", "")

    return (
        f"To,\n"
        f"The Grievance Officer\n"
        f"HDFC ERGO General Insurance Company Limited\n"
        f"Customer Happiness Center, D-301, 3rd Floor,\n"
        f"Eastern Business District, LBS Marg,\n"
        f"Bhandup (West), Mumbai - 400 078\n\n"
        f"Email: grievance@hdfcergo.com | care@hdfcergo.com\n\n"
        f"Subject: Policy No. {policy} — [Your Subject] | {name}\n\n"
        f"Dear Team,\n\n"
        f"[Letter body using context dict fields]\n\n"
        f"I request you to:\n"
        f"1. Acknowledge within 7 working days\n"
        f"2. Resolve within 15 working days\n\n"
        f"If not resolved, I reserve the right to escalate to the "
        f"Insurance Ombudsman, Bhubaneswar and file on IRDAI IGMS.\n\n"
        f"Yours faithfully,\n"
        f"[Policyholder Name]\n"
        f"Policy No: {policy}\n"
        f"Contact: [Phone] | [Email]"
    )
```

**Placeholder convention:**
- `[Policyholder Name]`, `[Phone]`, `[Email]` are intentional placeholders
- The user fills these in before sending
- Never auto-populate contact details from the schedule — the user must review before sending

**Connecting the letter to an insight:**
Pass the letter output to the `draft_letter` parameter of the `insight()` function:

```python
insights.append(insight(
    category="ACTION_REQUIRED",
    ...
    type="ACTION",
    priority="HIGH",
    draft_letter=draft_your_new_letter(sch, flag)
))
```

## Marking a Flag as Resolved

When an issue is corrected by the insurer, update the flag in Tier 2:

```json
{
  "flag_type": "SCHEDULE_INTEGRITY_ERROR",
  "status": "RESOLVED",
  "resolved_date": "2026-04-29",
  "resolution": "Description of what was corrected and how."
}
```

The engine checks `flag.get("status") == "RESOLVED"` before generating any insight or action letter from that flag. A RESOLVED flag is preserved in the JSON as institutional memory but produces no output.

**Never delete resolved flags.** They are a record of what happened and when.

## Adding a New Product to Tier 1

1. Create `product_library_{insurer}_{product_name}.json`
2. Ensure the following fields are present at minimum:
   - `_schema_meta.product_id` — unique identifier
   - `moratorium_period.months`
   - `waiting_periods.specific_disease_waiting`
   - `waiting_periods.ped_waiting`
   - `consumables_coverage`
   - `restore_benefit`
   - `optional_addons_available`
3. Create at least one user schedule (Tier 2) for this product
4. Run the full engine and verify all insights produce correct output
5. Add the product to `CURRENT_STATE.md` and update `README_V3.md` build status
